# Virtual Engineering Team Skills — 多角色 Agent 团队通用模式

> 来源：`garrytan/gstack` (MIT) — 23 角色虚拟工程团队设计
> 生成时间：2026-04-30
> 适用范围：任何需要多个 AI agent 协作完成复杂软件项目的工作流设计

## 一、核心思想：角色 ≠ 人，角色 = 质量闸门

gstack 的 23 个角色不是要模拟 23 个人，而是在软件开发的关键节点设置 **23 个质量检查点**。每个角色的本质是：
- **一个明确的职责边界**
- **一个结构化的审查清单**
- **一个产出物标准**

不迷信角色名，而应重视职责边界和检查清单。

## 二、核心角色与通用职责抽象

| 抽象角色 | gstack 对应 | 通用职责 | 何时调用 |

## 三、如何把角色转化为项目质量闸门

不要照搬 23 个角色名。按以下步骤设计你自己的质量闸门：

### Step 1：确定项目需要哪些闸门

```
Web 前端项目 → CEO + 设计审查 + 工程审查 + 代码审查 + 浏览器 QA + Ship
后端项目     → CEO + 工程审查 + 代码审查 + 安全审查 + QA + Ship
AI Agent 项目 → CEO + 工程审查 + 跨模型审查 + 安全审查 + Ship
小型脚本     → 代码审查 + Ship (简化)
```

### Step 2：为每个闸门定义检查清单

```
设计审查闸门示例：
□ 颜色/字体/间距一致性
□ 所有状态 (loading/empty/error/success) 都有 UI
□ 移动端响应式正常
□ 无 AI 视觉垃圾 (过度渐变、无意义动画)
```

### Step 3：建立闸门顺序

闸门是顺序的，不是并行的。上一个闸门通过才能进入下一个。

## 四、跨模型交叉验证模式

gstack 的重要创新：用两个不同模型 (Claude + Codex) 独立审查同一代码，生成交叉分析报告。

**价值：**
- 重叠发现 → 高置信度的真实问题
- 独有发现 → 需人工判断

**局限：**
- 两个模型一致 ≠ 绝对正确 (User Sovereignty 原则)
- 需要两个模型都有 API 访问权限
- 运营成本加倍

**迁移建议：**
如果只有一个模型可用，可以通过不同 prompt 策略模拟交叉视角：
- 审计模式：找 bug
- 建议模式：提改进
- 对抗模式：找最弱环节

## 五、不适用场景

- 个人小脚本 (不需要 23 个角色)
- 纯探索/研究 (不需要严格闸门)
- 每个闸门都需要 API 成本 (交叉验证需要多个 API key)
- 紧急 hotfix (跳过 Plan/Review，保留 Test/Ship)

## 七、关键原则

1. **User Sovereignty**：两个 AI 模型一致同意 ≠ 用户必须接受。AI 推荐，用户决定。
2. **Boil the Lake**：边际成本足够低时，做完整的事。不跳过测试、文档和边界情况。
3. **Search Before Building**：先搜后建。最坏的结果是完整实现了一个已有一行代码方案的东西。
